## 2.15 — CVS FAILURE MODES

### Failure Is Expected. Blockage Is Not.

Source: Jonathan M. Watson | 512 / CVS — Watson
Concept-ID: FAIL-OPEN-POSTURE · CVS-SIDECAR

CVS is designed to fail. Every system fails eventually.
The question is not whether CVS will experience
failure conditions but what happens when it does.

The answer is always the same: execution continues.
Evidence capture resumes when the failure condition
clears. The gap is visible. Nothing is hidden.

A CVS implementation that blocks execution on failure
is not CVS-conforming. It has violated Invariant 3
— exit must always be possible — and Invariant 6 —
on failure, systems must fail open. A witness layer
that holds the primary system hostage on failure
has become a control layer. That is not what CVS is.

### Failure Mode 1 — Sidecar Connectivity Loss

Source: Jonathan M. Watson | 512 / CVS — Watson
Concept-ID: CVS-SIDECAR · FAIL-OPEN-POSTURE

The sidecar loses connectivity to the execution
system's event streams. Events are still occurring
in the execution system but the sidecar is not
receiving them.

**What happens:**
Execution continues uninterrupted. The sidecar
buffers whatever events it can receive. When
connectivity is restored, evidence capture resumes
from the point of restoration.

**What is visible:**
A gap in the evidence chain at the point where
connectivity was lost. The gap is detectable by
any verifier examining the chain. The gap itself
is evidence — it proves that something interrupted
normal evidence capture at that moment.

**What is not acceptable:**
A sidecar that attempts to fill the gap retroactively
by reconstructing events from logs. Reconstruction
is not evidence capture. An Evidence Object assembled
from logs after the fact is a log entry, not a receipt.

### Failure Mode 2 — Ledger Unavailable

Source: Jonathan M. Watson | 512 / CVS — Watson
Concept-ID: CVS-SIDECAR · HASH-CHAINING

The settlement layer — the XRP Ledger or equivalent —
is temporarily unavailable. The sidecar is capturing
Evidence Objects but cannot anchor Merkle roots.

**What happens:**
Evidence capture continues normally. Evidence Objects
are generated, chained, and merkleised locally.
Merkle roots are queued for publication when the
ledger becomes available. When the ledger is restored,
queued roots are published in sequence.

**What is visible:**
A delay between the time events occurred and the
time their Merkle roots were anchored. This delay
is visible in the timestamps — the event timestamps
precede the anchor timestamps by the duration of
the outage.

**What is preserved:**
The internal integrity of the evidence chain is
maintained throughout. Hash chaining operates
independently of the settlement layer. The chain
remains tamper-evident even when it cannot be
externally anchored.

### Failure Mode 3 — Network Partition

Source: Jonathan M. Watson | 512 / CVS — Watson
Concept-ID: CVS-SIDECAR · EVIDENCE-OBJECT

The execution system and the sidecar are on different
network segments and the connection between them
is severed. The execution system continues operating.
The sidecar continues operating. They cannot see
each other.

**What happens:**
Execution continues. Evidence capture stops for
the duration of the partition. When the partition
is resolved, evidence capture resumes.

**What is visible:**
A gap in the evidence chain corresponding to the
duration of the partition. The gap has a defined
start point — the last Evidence Object before
partition — and a defined end point — the first
Evidence Object after restoration.

**The correct response:**
The gap is documented. It is not filled retroactively.
The organization notes in its operational record
that a network partition occurred during this window.
The gap, combined with the operational record,
provides a complete account of what happened —
more credible than a seamless record that conceals
a known interruption.

### Failure Mode 4 — Adversarial Behavior

Source: Jonathan M. Watson | 512 / CVS — Watson
Concept-ID: CVS-SIDECAR · GEOMETRIC-DISJOINT

An adversary — internal or external — attempts to
compromise the CVS evidence record. This could take
several forms:

**Attempt to modify existing Evidence Objects:**
Hash chaining makes this detectable. Any modification
changes the hash of the modified object, breaking
the link to the next object. The break is visible
to any verifier.

**Attempt to delete Evidence Objects:**
Deletion creates a gap in the chain. The gap is
visible. The Merkle root anchored to the settlement
layer covers the deleted objects. A gap between
the chain and the anchor proves deletion occurred.

**Attempt to inject false Evidence Objects:**
Injected objects would have different hashes from
those captured at execution time. The injection
breaks the chain at the point of injection. Additionally,
the injected objects cannot match the Merkle root
already anchored to the settlement layer for that
batch.

**Attempt to compromise the sidecar itself:**
If the sidecar is compromised, future Evidence Objects
may be unreliable. Past Evidence Objects remain valid —
they are already anchored and their integrity is
mathematically provable. The compromise affects
only future capture. The correct response is to
rotate keys, deploy a new sidecar instance, and
document the compromise window as a gap.

### What Failure Modes Prove

Source: Jonathan M. Watson | 512 / CVS — Watson
Concept-ID: CVS-SIDECAR · FAIL-OPEN-POSTURE

The treatment of failure modes reveals something
important about CVS design philosophy: visible
imperfection is preferable to invisible compromise.

A gap in the evidence record is honest. It says:
something interrupted evidence capture here. The
gap is informative. A verifier who sees a gap
knows to ask what happened during that window.

A seamless record that conceals an interruption
is dishonest — and more dangerous. It implies
continuous capture when capture was interrupted.
It creates false confidence in the completeness
of the record.

CVS is designed to be visibly imperfect rather
than invisibly compromised. That design choice
is what makes CVS evidence trustworthy — not
because it is always complete, but because its
incompleteness is always visible.
