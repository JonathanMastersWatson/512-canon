### What an Evidence Object Is

Source: Jonathan M. Watson | 512 / CVS — Watson
Concept-ID: EVIDENCE-OBJECT

An Evidence Object is the atomic unit of CVS output.
It is a self-contained, cryptographically sealed
record of a single execution event — produced at
execution time, by a witness external to the system
being described, in a form that can be verified by
any third party without trusting the operator.

The word receipt is more accurate than record.
A record describes. A receipt proves. An Evidence
Object proves that a specific event occurred under
specific conditions at a specific moment.

The distinction matters in adversarial contexts.
A record produced by the system under scrutiny
is evidence of what that system claims happened.
A receipt produced by an independent witness is
evidence of what happened. Courts, insurers, and
regulators treat these differently. CVS produces
the latter.

### The Four Required Elements

Source: Jonathan M. Watson | 512 / CVS — Watson
Concept-ID: EVIDENCE-OBJECT

Every Evidence Object must answer four questions.
An object that cannot answer all four is not an
Evidence Object. It is a log entry.

**What was proposed — intent hash**
A cryptographic hash of the proposed action at
the moment it was submitted for evaluation. This
proves what was asked, not what was permitted or
what occurred. The intent hash is captured at the
Pre-Validation observation point.

**What rules were active — spec hash**
A cryptographic hash of the constraint specification
that governed the evaluation. This proves which
rules applied at the moment of evaluation — not
which rules the organization claims applied, but
which rules were actually active in the system.

**What decision was made — per-invariant evaluation**
The result of evaluating the proposed action against
each invariant in the active constraint specification.
Not a summary pass/fail. A per-invariant record
showing which constraints were satisfied and which
were not. This proves the evaluation was real —
not a rubber stamp.

**What occurred — execution outcome**
The actual outcome: did the action execute, was it
refused, or was there a failure condition? This
proves what actually happened after the evaluation —
not what was authorized to happen.

### Self-Containment

Source: Jonathan M. Watson | 512 / CVS — Watson
Concept-ID: EVIDENCE-OBJECT · CVS-SIDECAR

Every Evidence Object must be independently sufficient.

A verifier must be able to confirm all four required
elements from a single Evidence Object without:
- Access to the execution system
- Access to internal logs
- Trust in the operator
- External context of any kind

If external context is required to interpret an
Evidence Object, it is not an Evidence Object.
It is a log reference — and log references require
trusting the system that produced the log.

Self-containment is not a convenience feature. It
is the property that makes CVS evidence usable in
the specific contexts where it matters most: legal
proceedings, regulatory examinations, and insurance
claims — all contexts where the operator of the
system under scrutiny cannot be trusted to provide
complete and accurate context.

### Formation Time

Source: Jonathan M. Watson | 512 / CVS — Watson
Concept-ID: EVIDENCE-OBJECT

Evidence Objects must be formed at execution time.

This requirement is absolute. An Evidence Object
assembled from logs after execution is not an
Evidence Object. It is a reconstruction — and
reconstructions are subject to all the limitations
that CVS is designed to overcome: incompleteness,
selective preservation, and the possibility of
manipulation by parties with access to the source
logs.

The formation time requirement means:
- The Pre-Validation record is created before
  evaluation begins
- The Validation Result record is created when
  evaluation completes
- The Post-Execution record is created when the
  action executes or is refused

No element is created afterward. No element is
assembled from other records. Each element exists
as an independent artifact the moment the event
it describes occurs.

### What Evidence Objects Are Not

Source: Jonathan M. Watson | 512 / CVS — Watson
Concept-ID: EVIDENCE-OBJECT

Evidence Objects are not:
- Audit trail reports generated periodically
- Log entries assembled after execution
- Compliance attestations produced by the system
  under review
- Behavioral profiles or analytics outputs
- System-generated explanations of decisions
- Reconstructions from partial records

A system that reconstructs evidence after execution
and describes the output as an Evidence Object is
not CVS-conforming. The reconstruction may be
accurate. It may be useful. It is not independent
evidence. It is the system's account of its own
behavior — which is precisely what CVS is designed
to replace.
