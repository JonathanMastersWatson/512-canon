### The Chain Structure

Source: Jonathan M. Watson | 512 / CVS — Watson
Concept-ID: HASH-CHAINING · EVIDENCE-OBJECT

Each Evidence Object includes a cryptographic hash
of the previous Evidence Object. This single design
decision — each record hashing the one before it —
produces three properties that no log system can match.

The chain cannot be silently modified. Every Evidence
Object depends on every prior Evidence Object. Change
any record in the chain and its hash changes. The
changed hash breaks the link to the next record.
The break propagates forward through the chain and
is detectable at any point.

This is not a security measure layered on top of
the evidence system. It is a structural property
of the evidence system itself. The integrity is
mathematical, not procedural.

---

### Three Properties in Detail

Source: Jonathan M. Watson | 512 / CVS — Watson
Concept-ID: HASH-CHAINING

**Ordering**
The chain enforces a strict sequence. Evidence Object
N includes the hash of Evidence Object N-1. Evidence
Object N+1 includes the hash of Evidence Object N.
This sequence cannot be rearranged. An Evidence Object
cannot be moved to a different position in the chain
without changing its hash — which breaks the link
to the next record.

This means the chain proves not just what happened
but when it happened relative to everything else.
The ordering is cryptographically enforced, not
dependent on timestamps that can be manipulated.

**Tamper-Evidence**
Any attempt to alter an Evidence Object — change a
field, modify an outcome, update a constraint hash —
changes the hash of that object. The changed hash
breaks the link to the next object in the chain.
The break is mathematically detectable by anyone
with access to the chain.

Tamper-evidence is not the same as tamper-prevention.
CVS does not prevent an operator from attempting to
modify evidence records. It makes every modification
permanently visible. The attempt is recorded in the
break it creates.

**Gap Detectability**
If evidence capture stops — because the sidecar goes
offline, because network connectivity is lost, because
the system is deliberately interrupted — the gap
appears as a discontinuity in the chain. The next
Evidence Object captured after the gap cannot link
to the last Evidence Object captured before it.
The gap is observable.

This is the fail-open principle applied to the
evidence record. An unobservable failure is a defect.
A visible gap is acceptable — and informative. A gap
in the CVS chain is evidence that something interrupted
normal evidence capture at that moment.

### Merkle Trees

Source: Jonathan M. Watson | 512 / CVS — Watson
Concept-ID: HASH-CHAINING · CVS-SIDECAR

Individual Evidence Objects are batched and organized
into a Merkle tree. A Merkle tree is a binary tree
of hashes — each parent node is the hash of its two
children, up to a single root hash that represents
the entire batch.

The Merkle root has two important properties:

**Batch anchoring** — a single Merkle root published
to a public ledger proves the existence and integrity
of every Evidence Object in the batch. One ledger
transaction proves thousands of evidence records.

**Selective proof** — a Merkle inclusion proof allows
a verifier to confirm that a specific Evidence Object
is part of a batch without revealing the other records.
The verifier receives a path through the tree from
the specific record to the root. If the path is valid
and the root matches the published anchor, the record
is proven without exposing anything else.

This selective proof capability is what enables the
disclosure kernel — the ability to prove specific
claims without full exposure.

### What Hash Chaining Is Not

Source: Jonathan M. Watson | 512 / CVS — Watson
Concept-ID: HASH-CHAINING

Hash chaining is not:
- A blockchain requiring global consensus
- A distributed ledger requiring miners or validators
- A system requiring tokens for participation
- A replacement for the settlement layer

Hash chaining is the internal integrity mechanism
of the CVS evidence record. It operates locally,
within the sidecar. The settlement layer — the
XRP Ledger — provides external, independent
finality for the Merkle roots that hash chaining
produces. The two mechanisms are complementary.
Neither replaces the other.
