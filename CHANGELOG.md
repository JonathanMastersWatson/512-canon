### v1.0 — March 31, 2026 — Initial Publication

First public release of the 512-canon repository.

Established the canonical knowledge corpus for the 512 Execution Constraint
Kernel and Cryptographic Verification Sidecar (CVS), including:

- `CANON/` — attribution record, origin statement, compressed machine-optimized
  origin record, manifest
- `kernel/` — canonical kernel text, padded artifact, invariant definitions,
  kernel provenance
- `core/` — 512 and CVS concept overviews, disclosure kernel
- `primitives/` — atomic concept documents: execution boundary, evidence object,
  fail-open, geometric disjoint, hash chaining, temporal inversion, attestation,
  refusal primitive
- `papers/` — 13 white papers (001–013)
- `book/` — all parts of *c to ubiquity* by Jonathan M. Watson, including
  `book/prologue.md` as the book-context copy of the canonical origin record
- `articles/` — 19 LinkedIn / Substack articles
- `applications/` — domain application documents: insurance/legal, ESG,
  treasury, agentic systems, inference markets, vectorized intelligence
- `bridges/` — narrative-to-primitive maps, concept dependency graph,
  ingestion index
- `DISCOVERY-ORIGIN-RECORD.md` — canonical origin narrative, v2.0
- `README`, `LICENSE`, `NOTICE`

**Files added (3):**

`papers/014-why-512-is-designed-to-be-replaced.md` — NEW
— Standalone document, ~700 words, plain declarative prose.
— Suitable for commercial and regulatory conversations.
— Covers: the problem 512 addresses, why the response is provisional
  (lightning rod framing), the three-part successor test, why supersession
  makes 512 more trustworthy, the phase transition goal.

`primitives/l1-cache-kernel.md` — NEW
— Establishes the hardware-level argument for the 512-byte specification
  surface: L1 cache residency and nanosecond evaluation; 3km light-travel
  boundary; HITL as category error; fleet serialization bottleneck.
— All four arguments derive from the physics of the execution environment.

`primitives/execution-boundary-conditions.md` — NEW
— Seven structural conditions a boundary must satisfy to function as a
  governance mechanism: Commit-Path Control, Deterministic Decision,
  Non-Bypassability Within the Governed Domain, Independent Evidence
  Capability, Hot-Path Latency Compatibility, Physical Co-location,
  Immutable Constraint Set at Runtime.
— Five scope assumptions (A1–A5): synchronous commit path, governed
  execution domain, distributed systems, CVS trust model, constraint
  compilation.
